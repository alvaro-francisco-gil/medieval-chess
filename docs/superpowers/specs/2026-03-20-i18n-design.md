# Internationalization (i18n) Design — English + Spanish

## Overview

Add full internationalization to Medieval Chess using **next-intl**, supporting English (default) and Spanish. Covers all UI strings, the rules page content, and page metadata. User-generated content (forum posts, puzzle descriptions) is excluded.

## Library Choice

**next-intl** — purpose-built for Next.js App Router, server-first, ICU MessageFormat, built-in middleware for locale routing and detection. Deployed on Vercel with zero extra config.

## Routing & URL Structure

- English (default, unprefixed): `medievalchess.com/rules`, `/play`, `/puzzles`, etc.
- Spanish (prefixed): `medievalchess.com/es/rules`, `/es/play`, `/es/puzzles`, etc.

### Middleware

A `middleware.ts` at the web app root handles:

1. **URL path check**: If `/es/*` → serve Spanish.
2. **Cookie check**: If user previously chose a language via the switcher, respect that.
3. **Browser detection**: On first visit with no cookie, check `Accept-Language` header. If browser prefers Spanish → redirect to `/es/*` and set cookie.
4. **Fallback**: English for unknown/unmatched languages.

### Locale config

- Supported locales: `["en", "es"]`
- Default locale: `"en"`
- Locale prefix strategy: `"as-needed"` (English has no prefix, Spanish gets `/es/`)

## Next.js Configuration

`apps/web/next.config.ts` must be wrapped with the next-intl plugin:

```ts
import createNextIntlPlugin from 'next-intl/plugin';
const withNextIntl = createNextIntlPlugin('./i18n/request.ts');
export default withNextIntl(nextConfig);
```

This wires up server-side message loading so `getTranslations` works.

### Pluralization

Use ICU MessageFormat for all plural forms:
```json

```

### Date and number formatting

Use next-intl's `useFormatter` hook (client) or `getFormatter` (server) for locale-aware date/number formatting. Replace hardcoded `toLocaleDateString("en-US", ...)` calls with the formatter.

### Chess-specific terms

- English: `"Queen (Alferza)"`, Spanish: `"Reina (Alferza)"` — Alferza appears in both languages.
- Same bilingual pattern for other historical terms (Elephant/Alfil, Grace Jump/Salto de Gracia).

### Translation scope

**Included:**
- All hardcoded UI strings (nav, buttons, headings, labels, placeholders, error messages)
- Full rules page content (~386 lines of explanations)
- Page metadata (`<title>`, `<meta description>`)
- Loading and error states
- Relative time strings (timeAgo)
- Difficulty labels
- Game status messages (checkmate, draw, turn indicator)
- Puzzle gameplay feedback strings

**Excluded:**
- User-generated content (forum posts, puzzle descriptions, comments)
- Chess notation (e4, Nf3, etc.)

  layout.tsx

  community/...

After:
```
app/
  layout.tsx          ← root layout: minimal, renders <html><body>{children}</body></html>, imports globals.css. Does NOT set lang attribute.
  not-found.tsx       ← root not-found: catches unmatched routes outside [locale]
  [locale]/
    layout.tsx        ← locale layout: sets <html lang={locale}>, wraps with NextIntlClientProvider + Providers/AuthProvider, renders NavBar
    not-found.tsx     ← locale not-found: translated 404 page
    page.tsx
    rules/page.tsx
    play/page.tsx
    puzzles/...
    community/...
    profile/page.tsx
```

### Root layout vs. locale layout

- **Root `app/layout.tsx`**: Minimal shell — imports `globals.css`, renders `<html>` and `<body>` without a `lang` attribute. Pass-through only.
- **Locale `app/[locale]/layout.tsx`**: Sets `<html lang={locale}>` via next-intl, wraps children with providers in this order:
  1. `NextIntlClientProvider` (outermost — so auth components can use translations)
  2. `Providers` / `AuthProvider` (inside next-intl provider)
  3. `NavBar` + page content

### Static generation

Each page under `[locale]/` must export `generateStaticParams` to enable static generation:
```ts
export function generateStaticParams() {
  return [{ locale: 'en' }, { locale: 'es' }];
}
```

## Component Integration

### Server Components

Most pages are server components. They use:
```tsx
import { getTranslations } from 'next-intl/server';

export default async function RulesPage() {
  const t = await getTranslations('rules');
  return <h1>{t('title')}</h1>;
}
```

Translations are resolved server-side and baked into HTML. Zero client bundle impact.

### Client Components

Interactive components (ChessBoard, auth flows) use:
```tsx
'use client';
import { useTranslations } from 'next-intl';

export function ChessBoard() {
  const t = useTranslations('play');
  return <button>{t('undo')}</button>;
}
```

The `[locale]/layout.tsx` wraps children with `NextIntlClientProvider`, passing messages for the current locale.

## Language Switcher

- Location: right side of NavBar, near auth buttons
- Style: simple "EN | ES" toggle, matching the medieval theme
- Behavior: navigates to the same page in the other locale using next-intl's `useRouter` and `usePathname` from `@/i18n/navigation`
- Persistence: next-intl middleware sets a `NEXT_LOCALE` cookie on language change

## SEO

### Hreflang tags

Every page includes in `<head>` via Next.js `metadata.alternates`:
```ts
export async function generateMetadata({ params }: Props): Promise<Metadata> {
  const { locale } = await params;
  const t = await getTranslations({ locale, namespace: 'rules' });
  return {
    title: t('metaTitle'),
    description: t('metaDescription'),
    alternates: {
      canonical: locale === 'en' ? '/rules' : '/es/rules',
      languages: {
        en: '/rules',
        es: '/es/rules',
        'x-default': '/rules',
      },
    },
  };
}
```

### Localized metadata

Each page generates locale-specific `<title>` and `<meta name="description">` via Next.js `generateMetadata()` using translations.

## Auto-detection Flow

1. User visits `medievalchess.com/` for the first time (no cookie).
2. Middleware reads `Accept-Language` header.
3. If Spanish is preferred → redirect 307 to `/es/` and set `NEXT_LOCALE=es` cookie.
4. If English or unknown → serve English, set `NEXT_LOCALE=en` cookie.
5. On subsequent visits, cookie takes priority over browser header.
6. Manual language switch via toggle overrides the cookie.
